  • CMMI还是Agile

    22 April, 2015

    开篇...

    软件项目管理到底用CMMI还是Agile呢？最近参加的各种活动会议毫无疑问推广的都是Agile，但是，实时上Agile是万能的吗？任何公司或者团队都可以照搬吗？事实证明Agile并不容易实施，失败的案例数不胜数。下面以我自己的观点分别分析一下这两种模式。

    CMMI...

    最著名的传统式软件开发莫过于V模型了，下面一张图摘自网络，很好的诠释了V模型的原理：

    整体结构非常公正，一环扣一环，层层把关，所以经过这样层层把关出来的东西有可能不完美，但是很少出现根本没法用的情况（当然如果你完全不了解V模式的合作方式另论）。说说整个项目中的人员吧：business，BSA，AD，PM，TL, Developers, Testers, Integration Team, Maintainance Team，有的还有PMO（or PMC）等等。有这么庞大的团队，功能职责这么全，很少出现出来的东西不能用的情况。

    那是不是这样一种模式就是完美的呢？当然不是了，否则也不会出现敏捷模式的风靡了。归根到底在于这个团队太过于庞大，就如千军万马行动迟缓，一个改动要经过层层审批测试，一般这样的项目要一个季度以上才能发布一个release，而且个人的能力也比较难于得到体现。想想看我们另一个耳熟能详的模式原型模式，为什么传统开发非常少用这种模式呢？因为周期太长了，如果采用原型，肯定会延缓项目进度。

    Agile...

    Agile的流行也有一段时间了，很多人都把敏捷奉若神明一样，什么样的人，什么样的团队都说自己是敏捷的，感觉敏捷就是高大上，能用少量的人解决一切问题。然而大家注意到没有，敏捷失败的案例非常的多。大家注意到的是敏捷的短迭代，没有那么多文档，没有那么多测试，所以开发团队很开心的在开发新功能，不太关心文档，不太关心测试，结果可想而知。

    优点

    1. 敏捷讲究的是迅速，也就是对时间的把控，所以用好了肯定要比传统开发快。
    2. 敏捷对纪律性有一定的要求，要经常性的检查进度，这个也是我认为它效率高的主要原因。
    3. 敏捷对成员有一定的要求，后续的检查没那么多，基本代码写定了，软件质量也就定了。
    4. 敏捷虽然不要求那么多文档，但是至少要有一个大而全的文档，否则也不会出来那么多敏捷的软件了。
    5. 敏捷也要跟踪，便于检查问题总结经验。

    缺点

    1. 敏捷对团队的能力有较高的要求，早期定位那些功能方向很重要。
    2. 对团队要求高的另一点是简化的流程（集成，测试等等）并不是没有了，而是自动化了。
    3. 如果有维护需求的项目容易导致进度难于控制（感觉这个算是真正的缺点）。
    4. 如果每天对团队成员的时间管理过于紧张，容易引发负面情绪。

    留个小问题思考

    我们听过的敏捷团队往往都很小的团队，如果你的项目很大，没法通过小团队来完成，需要多个团队协同完成，那怎么去敏捷这个大的团队呢？

    总结...

    最近参与了IBM的敏捷活动，发现某些公司完全没有章程的开发才是最大的问题，自上而下完全不懂软件工程，人员能力比较一般，对上埋怨领导想到啥提啥，对下埋怨工作进度质量差。跟自己的工作经历结合，有感而发，有些未必对。

    总之，采取那么开发模式最好对软件工程有一些认知，选用那些软件管理项目，根据自身的成本考虑，没有说收费的就一定比免费的好，所以选择对的才是最重要。而且我比较反对敏捷采用8小时Charge的方式，压力太大。

  • My First Blog to use GitHub

    20 April, 2015

    About github...

    Github is really famous now, not only for open source, but also for static blogs.

    And this is the first time for me to try to use github to publish my blogs.

    I have tried some static blog projects such as hexo (made with node.js), and hugo (made with golang), of course the first try is jekyll (made with Ruby), but at the end I still choose pretzel(made by C#). The reason why I choose this because I am familiar with C#, and also it has a wonderful IDE.

    Usage

    The principal commands are the following but more informations are available on the wiki

    Create is used to create the folder structure for a new site.

    pretzel create

    If the site should be at a specific folder, this can be specified as a parameter:

    pretzel create C:\path\to\folder

    Bake is used to generate a site based on the contents of a folder.

    To scan the current directory for a website and detect the content to process, run:

    pretzel bake

    To scan a specific folder and parse the contents, run:

    pretzel bake C:\path\to\folder

    To explicitly specify the input - we should support inferring the input based on the files found anyway - run:

    pretzel bake --engine liquid

    Taste is for testing a site locally - make a change, and pretzel should handle regenerating the page when a file changes.

    To test a site locally, run:

    pretzel taste

    To specify the port to serve the site from (default will be 4000), run:

    pretzel taste --port 5000

    To stop the site, press q.
